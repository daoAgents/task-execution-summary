## 1. 文档版本控制规范

### 1.1 版本号格式

版本号采用**主版本号.次版本号**格式：

- **标准格式**: `vX.Y`（如 v2.0、v2.1）
- **Quick版格式**: `vX.Y-quick`（如 v2.0-quick）
- **示例**: v2.0、v2.1、v2.0-quick

### 1.2 版本号与功能版本对齐

| 场景 | 版本号变化 | 示例 |
|------|-----------|------|
| 功能新增（向后兼容） | 次版本号+1 | v2.0 → v2.1 |
| 破坏性变更 | 主版本号+1 | v2.5 → v3.0 |
| 内容修正（错别字、格式） | 次版本号+1 | v2.0 → v2.1 |

### 1.3 版本号层级关系

```
SKILL.md (v2.5)
    │
    ├── P0层: 4个quick文件 (v2.0-quick)
    │
    ├── P1层: 5个完整参考文档 (v2.0)
    │
    ├── P3层: terminology.md (v2.1)
    │
    └── evals/: evals.json (v1.0)
```

**核心原则**: SKILL.md作为主文档，版本号应领先或等于references/下的文档版本号。

### 1.4 Quick版与完整版版本同步

- Quick版和完整版必须保持**同主版本号**
- Quick版通过后缀`-quick`标识
- 完整版更新时，Quick版必须同步评估是否需要更新

## 2. 文档更新流程

### 2.1 标准化更新流程

```
变更申请 → 影响评估 → 审核 → 实施 → 验证 → 发布
```

| 阶段 | 关键动作 | 输出物 |
|------|---------|--------|
| 变更申请 | 记录变更原因、范围、预期效果 | 变更申请单 |
| 影响评估 | 识别关联文档、评估Token影响 | 影响评估报告 |
| 审核 | 审查变更必要性、版本号合理性 | 审核意见 |
| 实施 | 修改文档内容、更新版本号 | 修改后的文档 |
| 验证 | 执行一致性检查、验证引用准确性 | 验证报告 |
| 发布 | 更新SKILL.md版本对照表、归档旧版本 | 发布记录 |

### 2.2 小修流程（简化）

适用于错别字、格式调整、链接修正等低风险变更：

1. 直接修改目标文档
2. 更新文档底部变更记录
3. 更新SKILL.md中的版本对照表（如版本号变化）
4. 完成一致性检查清单（简化版）

### 2.3 大改流程（完整）

适用于功能变更、结构调整、新增章节等高风险变更：

1. **变更申请**: 提交变更申请单，说明变更原因和范围
2. **影响评估**: 识别所有受影响文档（包括Quick版/完整版对应关系）
3. **审核**: 由维护者审核变更必要性和版本号变更合理性
4. **实施**: 按优先级顺序更新文档（P0 → P1 → P3）
5. **验证**: 执行完整一致性检查
6. **发布**: 更新SKILL.md版本对照表，记录变更日志

## 3. 版本漂移防控

### 3.1 版本漂移定义

版本漂移指文档间出现以下不一致情况：

- 功能描述与实际实现不符
- 跨文档引用的版本号不一致
- Quick版与完整版内容不同步
- SKILL.md中的Token数与实际文件不符
- 错误码数量统计不一致

### 3.2 防控规则

#### 规则1: 功能变更同步更新

功能变更必须同步更新以下文档：

| 变更类型 | 必须更新的文档 |
|----------|---------------|
| API变更 | api-reference.md + api-reference-quick.md |
| 错误码变更 | error-codes.md + error-codes-quick.md |
| 流程变更 | execution-flow.md + execution-flow-quick.md |
| 模板变更 | templates.md + templates-quick.md |

#### 规则2: Quick版与完整版同步

- 完整版新增/修改内容，评估Quick版是否需要同步
- Quick版作为完整版的摘要，关键信息必须一致
- 禁止Quick版与完整版出现矛盾描述

#### 规则3: SKILL.md引用准确性

SKILL.md中的以下信息必须与实际文件一致：

- 文档版本号
- Token预算数据
- 功能描述
- 加载优先级标注

#### 规则4: 统计数据定期校准

project-review-report.md中的统计数据需定期校准：

| 数据项 | 校准频率 | 校准方法 |
|--------|---------|---------|
| 各文件Token数 | 每月 | 重新统计并对比 |
| 错误码数量 | 每次错误码变更 | 核对error-codes.md |
| 文件大小/行数 | 每季度 | 自动化脚本统计 |

## 4. 一致性检查机制

### 4.1 检查清单

#### 跨文档引用完整性检查

- [ ] 所有文档内链接可正常跳转
- [ ] 引用的文档版本号存在且有效
- [ ] 术语引用与terminology.md一致

#### 版本号一致性检查

- [ ] SKILL.md版本对照表与实际文档版本一致
- [ ] Quick版与完整版主版本号一致
- [ ] 变更记录中的版本号变化逻辑正确

#### Token数据准确性检查

- [ ] SKILL.md中Token预算与实际文件大小匹配
- [ ] project-review-report.md中Token统计准确
- [ ] 场景化加载推荐中的Token预估合理

#### 错误码数量一致性检查

- [ ] SKILL.md中提及的错误码数量与error-codes.md一致
- [ ] error-codes-quick.md中的错误码列表与完整版一致

### 4.2 检查频率

| 检查类型 | 触发条件 | 执行人 |
|----------|---------|--------|
| 即时检查 | 每次文档变更后 | 变更实施者 |
| 月度检查 | 每月最后一个工作日 | 维护负责人 |
| 季度检查 | 每季度末 | 架构委员会 |

### 4.3 检查范围

参考project-review-report.md中已识别的关联关系：

| 文档 | 关联文档 | 关联类型 |
|------|---------|---------|
| SKILL.md | 全部references/文档 | 引用、版本对照 |
| api-reference.md | api-reference-quick.md, examples-v2.md | 内容对应、示例引用 |
| error-codes.md | error-codes-quick.md | 内容摘要 |
| execution-flow.md | execution-flow-quick.md | 内容摘要 |
| templates.md | templates-quick.md | 内容摘要 |
| examples-v2.md | api-reference.md | 示例代码引用 |

## 5. 维护责任制度

### 5.1 文档分层与维护优先级

| 层级 | 包含文档 | 维护优先级 | 更新策略 |
|------|---------|-----------|---------|
| **P0** | SKILL.md + 4个quick文件 | 最高 | 每次功能变更必须同步更新 |

| **P3** | terminology.md + v3-planning/ | 中 | 季度审查 |

### 5.2 维护责任人

| 文档/层级 | 责任人 | 职责 |
|-----------|--------|------|
| P0层文档 | 核心维护团队 | 确保日常加载文档的准确性和一致性 |
| P1层文档 | 功能模块负责人 | 维护各自模块的详细文档 |

| 版本对照表 | 核心维护团队 | 统一维护SKILL.md中的版本对照表 |

### 5.3 更新响应时间

| 变更类型 | P0层 | P1层 | P3层 |
|----------|------|------|------|
| 紧急修复 | 24小时内 | 48小时内 | 下次季度审查 |
| 功能更新 | 同步更新 | 同步更新 | 下次季度审查 |
| 定期优化 | 每月 | 每季度 | 每半年 |

## 6. 变更日志规范

### 6.1 变更记录格式

每个文档底部必须维护变更记录，格式如下：

```
变更记录：vX.Y→vX.Z - 变更描述
```

**多行变更记录示例**（参考SKILL.md）：

```
*变更记录：v2.0→v2.1 - Description触发能力优化（基于20个eval查询分析，覆盖率60%→100%）*
*变更记录：v2.1→v2.2 - 强化分层加载规则，更新Token预算（P1层优化后从~50.8K降至~44.4K）*
*变更记录：v2.2→v2.3 - P2优化：场景化加载规则、Token预算校准、Quick/完整版互斥规则、冗余精简*
```

### 6.2 详细变更记录格式

对于重大版本变更，参考api-reference.md第6节的版本历史格式：

```markdown
### vX.Y (YYYY-MM-DD) - 版本标题

**发布日期**: YYYY年M月D日

**版本类型**: Major/Minor Release

**新增功能**：
- 功能点1
- 功能点2

**修复问题**：
- 问题1
- 问题2

**已知限制**：
- 限制1

**兼容性说明**：
- 兼容性描述
```

### 6.3 变更记录要素

每条变更记录必须包含：

| 要素 | 说明 | 示例 |
|------|------|------|
| 版本号变化 | 变更前后的版本号 | v2.0→v2.1 |
| 变更日期 | 变更实施日期 | 2026-04-15 |
| 变更内容摘要 | 简要描述变更内容 | Description触发能力优化 |
| 影响范围 | 影响的文档或功能范围 | 基于20个eval查询分析 |

## 7. 执行保障机制

### 7.1 文档变更Checklist

每次变更前必须完成以下检查：

#### 变更前检查

- [ ] 已明确变更范围和原因
- [ ] 已识别所有受影响文档
- [ ] 已确定版本号变更方案
- [ ] 已评估Token影响（如涉及）

#### 变更中检查

- [ ] 按优先级顺序更新文档（P0 → P1 → P3）
- [ ] Quick版与完整版同步更新
- [ ] 文档内链接有效性验证
- [ ] 变更记录已添加

#### 变更后检查

- [ ] SKILL.md版本对照表已更新
- [ ] 跨文档引用一致性已验证
- [ ] Token统计数据已校准（如涉及）
- [ ] 变更已记录到变更日志

### 7.2 定期审查制度

#### 月度审查

| 审查项 | 审查内容 | 输出物 |
|--------|---------|--------|
| 版本一致性 | 核对所有文档版本号 | 一致性报告 |
| Token统计 | 验证Token预算准确性 | Token统计表 |
| 变更记录 | 检查变更记录完整性 | 变更汇总 |

#### 季度审查

| 审查项 | 审查内容 | 输出物 |
|--------|---------|--------|
| P3层文档 | 术语表和V3规划文档审查 | 审查报告 |
| 加载策略 | 评估分层加载策略有效性 | 优化建议 |
| 版本漂移 | 全面检查版本漂移情况 | 漂移检测报告 |

### 7.3 版本漂移修复流程

发现版本漂移后的修复流程：

```
发现漂移 → 影响评估 → 制定修复方案 → 实施修复 → 验证 → 复盘
```

| 阶段 | 关键动作 |
|------|---------|
| 发现漂移 | 通过一致性检查或用户反馈发现版本不一致 |
| 影响评估 | 评估漂移影响范围和严重程度 |
| 制定修复方案 | 确定修复优先级和具体步骤 |
| 实施修复 | 按优先级修复不一致项 |
| 验证 | 执行完整一致性检查 |
| 复盘 | 分析漂移原因，制定预防措施 |

### 7.4 漂移严重程度分级

| 级别 | 定义 | 响应时间 | 处理策略 |
|------|------|---------|---------|
| Critical（严重） | 功能描述错误、版本号矛盾 | 24小时内 | 立即修复 |
| Medium（中等） | Token数据偏差、链接失效 | 72小时内 | 优先修复 |
| Minor（轻微） | 格式不一致、描述不精确 | 下次定期审查 | 批量修复 |
