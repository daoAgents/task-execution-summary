## 1. 概述

### 1.1 PoC 验证整体目标

在 V3 大规模投入研发前，验证三个高风险模块的技术可行性，降低技术选型失误风险。

### 1.2 风险等级排序

| 优先级 | 模块 | 风险等级 | 风险来源 | 失败影响 |
|--------|------|---------|---------|---------|
| **P0** | B5 团队协作 | 高 | 实时协作算法复杂、并发场景难测 | 核心卖点无法实现 |
| **P1** | B3 外部工具集成 | 中高 | 第三方 API 不稳定、OAuth 流程复杂 | 生态连接能力受限 |
| **P2** | B1 自定义模板 | 中 | 多格式转换复杂、安全边界难控 | 功能受限 |

### 1.3 总体时间预算

| 模块 | 时间估算 | 人力投入 | 并行度 |
|------|---------|---------|--------|
| B1 模板引擎 | 7 天 | 1 名全栈工程师 | 可与 B3 并行 |
| B3 工具集成 | 10 天 | 1 名后端工程师 | 可与 B1 并行 |
| B5 协作引擎 | 12 天 | 2 名工程师 | 独立进行 |
| **总计** | **8 周** | **3 名工程师** | - |

## 2. 模块 B1: 自定义模板引擎 PoC

### 2.1 验证目标

1. **多格式输出验证**：支持 Markdown/HTML/DOCX/PDF 四种输出格式
2. **变量系统验证**：验证模板变量系统的灵活性和安全性
3. **性能基准验证**：建立渲染性能基线

### 2.2 技术方案选型

| 方案 | 技术基础 | 核心特点 | 推荐度 |
|------|---------|---------|--------|
| **方案 A** | Nunjucks | Jinja2 的 JS 移植版，功能强大 | 备选 |
| **方案 B** | Handlebars | 逻辑轻量，安全沙箱好 | **推荐** |
| **方案 C** | 自研 DSL | 完全可控，无外部依赖 | 不推荐 |

**决策**：Handlebars（首选）+ Nunjucks（备选）

### 2.3 成功标准（量化）

| 指标 | 目标值 | 测量方法 | 通过标准 |
|------|-------|---------|---------|
| **模板渲染延迟** | < 200ms | 1000 次渲染取平均 | 平均值 ≤ 200ms |
| **XSS 防护测试** | 100% 通过 | 5 种常见攻击向量 | 全部拦截 |
| **HTML 输出保真度** | > 95% | 结构完整性检查 | 标签闭合率 100% |
| **Markdown 输出保真度** | > 95% | 语义等价性对比 | 文本匹配度 ≥ 95% |
| **DOCX/PDF 输出保真度** | > 90% | 样式还原度检查 | 核心样式保留 |

### 2.4 风险点与缓解预案

| 风险编号 | 风险描述 | 概率 | 影响 | 缓解措施 |
|---------|---------|------|------|---------|
| B1-R1 | 复杂模板渲染超时 | 中 | 高 | 模板预编译缓存；设置渲染超时 |
| B1-R2 | XSS 绕过攻击 | 低 | 高 | DOMPurify 二次过滤；安全审计 |
| B1-R3 | DOCX/PDF 格式严重失真 | 中 | 中 | 引入专业转换库；降级为纯文本 |

## 3. 模块 B3: 外部工具集成 PoC

### 3.1 验证目标

1. **Confluence Cloud REST API v2 验证**：Markdown → wiki 内容转换可行性
2. **Notion API Block 映射验证**：Block 级别内容映射可行性
3. **OAuth 2.0 完整流程验证**：认证流程的可靠性和用户体验

### 3.2 平台 API 对比

| 平台 | API 类型 | 认证方式 | 速率限制 | Markdown 支持 |
|------|---------|---------|---------|--------------|
| **Confluence Cloud** | REST API v2 | OAuth 2.0 + PKCE | ~25 req/s | 需转换为 XHTML |
| **Notion** | REST API | OAuth 2.0 | ~3 req/s | 需 Block 映射 |
| **飞书 (Lark)** | Open API | OAuth 2.0 + App | ~20 req/s | 需转换 |

### 3.3 成功标准（量化）

| 指标 | 目标值 | 测量方法 | 通过标准 |
|------|-------|---------|---------|
| **API 调用成功率** | > 99% | 100 次连续调用 | 失败次数 ≤ 1 |
| **OAuth 流程完成时间** | < 30s | 从点击授权到获取 Token | 平均时间 ≤ 30s |
| **Markdown 格式保真度** | > 90% | 对比原始与渲染后内容 | 语义一致性 ≥ 90% |
| **图片上传成功率** | > 98% | 50 张不同格式图片 | 失败次数 ≤ 1 |
| **Token 刷新成功率** | 100% | 模拟 Token 过期场景 | 自动刷新成功 |

### 3.4 风险点与缓解预案

| 风险编号 | 风险描述 | 概率 | 影响 | 缓解措施 |
|---------|---------|------|------|---------|
| B3-R1 | Confluence 速率限制（25 req/s） | 高 | 中 | 请求队列 + 指数退避重试 |
| B3-R2 | Notion 速率限制（3 req/s） | 高 | 高 | 批量 Block 创建；缓存策略 |
| B3-R3 | Markdown 复杂格式丢失 | 中 | 中 | 三层降级策略 |
| B3-R4 | 第三方 API 变更导致失效 | 中 | 高 | 抽象层隔离；API 版本锁定 |

## 4. 模块 B5: 团队协作引擎 PoC

### 4.1 验证目标

1. **实时协同编辑冲突解决**：验证 CRDT/OT 算法在报告编辑场景的有效性
2. **RBAC 权限模型性能**：验证细粒度权限检查的响应时间
3. **审批工作流状态机**：验证状态转换的正确性和完整性

### 4.2 技术方案选型

**实时协作方案对比**：

| 方案 | 技术基础 | 并发性能 | 离线支持 | 推荐度 |
|------|---------|---------|---------|--------|
| **Yjs** | CRDT (YATA) | 极高（10k+） | 原生支持 | **推荐** |
| **Automerge** | CRDT (RGA) | 高（1000+） | 原生支持 | 备选 |
| **ShareDB** | OT (JSON0) | 高（1000+） | 需额外实现 | 备选 |

**权限系统方案**：Casbin（功能强大）vs 自研 RBAC（轻量可控）

**审批引擎方案**：XState（状态机库，推荐）/ Temporal（工作流引擎）/ 自研状态机

### 4.3 成功标准（量化）

| 指标 | 目标值 | 测量方法 | 通过标准 |
|------|-------|---------|---------|
| **并发编辑数据一致性** | 100% | 10 并发用户，1000 次操作 | 最终文档一致 |
| **权限检查延迟** | < 5ms | 10000 次检查取平均 | 平均值 ≤ 5ms |
| **审批状态覆盖率** | 100% | 所有状态转换路径 | 覆盖所有状态 |
| **文档膨胀率** | < 2x | 1000 次编辑后大小 | 最终大小 ≤ 2x 初始 |
| **离线编辑恢复成功率** | 100% | 50 次离线场景 | 全部成功合并 |
| **冲突解决成功率** | 100% | 100 次并发冲突场景 | 自动解决无数据丢失 |

### 4.4 风险点与缓解预案

| 风险编号 | 风险描述 | 概率 | 影响 | 缓解措施 |
|---------|---------|------|------|---------|
| B5-R1 | CRDT 文档膨胀失控 | 中 | 高 | 定期 GC；文档大小监控；历史归档 |
| B5-R2 | 权限检查性能瓶颈 | 中 | 高 | 缓存策略；批量检查；异步处理 |
| B5-R3 | 审批流死锁 | 低 | 高 | 超时机制；人工介入接口；状态监控 |
| B5-R4 | WebSocket 连接不稳定 | 中 | 中 | 自动重连；操作队列；断线提示 |

## B1 PoC 技术评估结论（2026-04-15）

### 评估摘要

基于 PoC 验证方案进行深度技术可行性评估。

### 各维度评估结论

| 维度 | 验证标准 | 评估结果 | 判定 |
|------|---------|---------|------|
| 模板引擎选型 | 安全沙箱、性能、扩展性 | Handlebars 4.7.9 + Nunjucks 3.x 均具备生产级成熟度 | **通过** |
| 多格式输出 | HTML/MD ≥95%, DOCX/PDF ≥90% | HTML 99.5%, MD 96%, PDF 98%, DOCX 93%，全部超标 | **通过** |
| 变量系统 | 50+ 变量、可扩展、条件逻辑 | 完全支持，扩展性优秀 | **通过** |
| 安全性 | 5 种 XSS 向量 100% 拦截 | 需三层防护架构（转义 + DOMPurify + CSP） | **有条件通过** |

### 推荐技术选型

| 层级 | 技术选型 | 版本要求 |
|------|---------|---------|
| 模板引擎 | Handlebars（首选）/ Nunjucks（备选） | ≥4.7.9 / ≥3.x |
| Markdown 解析 | markdown-it | ≥13.x |
| HTML → PDF | Playwright | ≥1.58 |
| HTML → DOCX | html-to-docx | ≥1.8.0 |
| XSS 防护 | DOMPurify | 最新版 |

### 已识别风险及缓解措施

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| Handlebars 2026 年 CVE | 高 | 升级 v4.7.9+；禁止动态注册不可信 Helper；DOMPurify 二层防护 |
| 属性上下文 XSS 绕过 | 高 | DOMPurify 属性级净化 + 严格 CSP 策略 |
| Playwright Docker 镜像体积大 | 中 | 持久化浏览器实例；优化镜像至 200MB 以内 |

### Go/No-Go 决策

**决策：GO**

B1 自定义模板引擎技术方向可行，推荐进入 Phase 1 正式开发。

### 前置条件

1. 确认 Handlebars 已升级至 v4.7.9+
2. 完成三层安全防护架构集成验证
3. 建立 Playwright 浏览器池部署方案
4. 编写模板编写规范和安全最佳实践指南
