## 1. 调研概述

### 1.1 调研背景与目的

#### 1.1.1 背景
V3版本规划包含五大核心模块（B1-B5），但各模块的真实用户需求强度、优先级排序及商业化潜力尚未经过系统性验证。为避免资源错配，需在开发启动前完成定量+定性混合调研，为产品路线图提供数据支撑。

#### 1.1.2 核心研究问题

| 优先级 | 研究问题 | 对应模块 |
|--------|----------|----------|
| P0 | B1-B5各功能模块的真实需求强度如何排序？ | B1-B5 |
| P0 | 不同用户群体对五大模块的需求是否存在显著差异？ | B1-B5 |
| P1 | 用户的付费意愿强度及可接受价格区间？ | 商业化 |
| P1 | 当前竞品在五大功能领域的覆盖现状？ | 竞品分析 |
| P2 | 用户当前未被满足的痛点有哪些？ | 机会挖掘 |

#### 1.1.3 调研目标

1. **验证假设**：量化验证V3五大模块的需求优先级假设
2. **用户分层**：建立基于角色、规模、使用频率的三维用户分层模型
3. **商业洞察**：评估付费意愿，为定价策略提供依据
4. **竞品对标**：识别竞品功能缺口，明确差异化机会点

### 1.2 调研方法论

#### 1.2.1 混合研究设计

采用**解释性时序设计(Explanatory Sequential Design)**：

```
第一阶段(定量) → 第二阶段(定性) → 整合分析
   问卷调查      →   深度访谈      →   三角验证
  (N=400+)     →   (N=12-15)    →   决策建议
```

#### 1.2.2 三角验证原则

| 验证维度 | 定量数据 | 定性数据 | 交叉验证点 |
|----------|----------|----------|------------|
| 需求强度 | Likert量表评分 | 访谈中的痛点描述 | 高分是否对应高频提及 |
| 优先级 | 排序题结果 | 资源分配假设 | 排序一致性检验 |
| 付费意愿 | 价格敏感度测试 | 预算决策流程 | 意愿与能力匹配度 |

## 3. 定量调研：结构化问卷

### 3.1 问卷设计原则

#### 3.1.1 设计规范

| 原则 | 说明 | 实施方式 |
|------|------|----------|
| 漏斗结构 | 从泛到精，降低认知负荷 | 基本信息→功能需求→竞品→付费→开放 |
| 逻辑跳转 | 根据答案动态调整后续题目 | 如"未使用过"则跳过满意度题 |
| 随机排序 | 避免顺序偏差 | 选项随机、矩阵题行随机 |
| 注意力检测 | 筛选无效问卷 | 插入1道注意力检测题 |
| 长度控制 | 完成时间5-8分钟 | 18-20题，单屏显示 |

#### 3.1.2 量表说明

| 量表类型 | 应用场景 | 选项定义 |
|----------|----------|----------|
| 5级Likert | 需求强度、满意度 | 1=完全不需要/非常不满意，5=非常需要/非常满意 |
| 7级Likert | 价格敏感度 | 1=绝对不接受，7=非常愿意接受 |
| NPS量表 | 推荐意愿 | 0-10分，0-6贬损者，7-8被动者，9-10推荐者 |

## 5. 样本量与抽样策略

### 5.1 统计学最小样本量计算

#### 5.1.1 定量问卷样本量

**计算公式：**
```
n = (Z² × p × (1-p)) / e²
```

**参数说明：**
| 参数 | 值 | 说明 |
|------|-----|------|
| Z | 1.96 | 95%置信水平对应Z值 |
| p | 0.5 | 预估比例（取最保守值） |
| e | 0.05 | 允许误差5% |

**计算结果：**
```
n = (1.96² × 0.5 × 0.5) / 0.05²
n = (3.8416 × 0.25) / 0.0025
n = 0.9604 / 0.0025
n = 384.16 ≈ 385
```

**考虑无效问卷（预估10%无效）：**
```
目标样本 = 385 / 0.9 = 428
向上取整 = 450（最小有效样本）
建议目标 = 500（预留分析余量）
```

## 6. 数据分析框架

### 6.1 定量分析方法

#### 6.1.1 描述性统计

| 统计指标 | 应用场景 | 输出形式 |
|----------|----------|----------|
| 频数分布 | 单选题、多选题 | 柱状图、饼图 |
| 均值/中位数 | Likert量表 | 表格、雷达图 |
| 标准差 | 数据离散程度 | 误差线 |
| 交叉频数 | 多变量关系 | 交叉表、热力图 |

#### 6.1.2 交叉分析维度

| 分析维度 | 交叉变量 | 分析目的 |
|----------|----------|----------|
| 角色×需求 | 角色 vs B1-B5评分 | 不同角色的需求差异 |
| 规模×付费 | 公司规模 vs 付费意愿 | 企业级市场潜力 |
| 频率×功能 | 使用频率 vs 功能偏好 | 重度用户特征 |
| 工具×满意度 | 当前工具 vs 各维度满意度 | 竞品弱点识别 |

#### 6.1.3 统计检验

| 检验方法 | 应用场景 | 假设 |
|----------|----------|------|
| 卡方检验 | 分类变量独立性 | 角色与功能偏好是否独立 |
| t检验 | 两组均值比较 | 付费vs免费用户的功能需求差异 |
| 方差分析(ANOVA) | 多组均值比较 | 不同规模企业的需求差异 |
| 相关性分析 | 变量关联程度 | 使用频率与付费意愿的相关性 |

### 6.2 功能优先级加权评分公式

#### 6.2.1 评分模型

```
Module_Score = (Demand × 0.40) + (Willingness × 0.25) + (Differentiation × 0.20) + (Feasibility × 0.15)
```

#### 6.2.2 指标定义与计算

| 指标 | 权重 | 数据来源 | 计算方法 |
|------|------|----------|----------|
| **Demand** 需求强度 | 40% | B1题Likert评分 | 模块平均评分 / 5 × 100 |
| **Willingness** 付费意愿 | 25% | D1题+D3题 | 付费意愿系数 × 价格接受度 |
| **Differentiation** 差异化价值 | 20% | C1题满意度+C2切换意愿 | (5-竞品满意度均值) × 切换意愿系数 |
| **Feasibility** 技术可行性 | 15% | 技术团队评估 | 主观评分1-5标准化 |

#### 6.2.3 计算示例

**假设B1自定义模板模块的数据：**

| 指标 | 原始数据 | 标准化得分 |
|------|----------|------------|
| Demand | Likert均值4.2 | 4.2/5×100 = 84 |
| Willingness | 65%愿意付费，均价￥60 | 65 × 0.8 = 52 |
| Differentiation | 竞品满意度3.2，切换意愿3.8 | (5-3.2) × 0.76 = 68.4 |
| Feasibility | 技术评估4.0 | 4.0/5×100 = 80 |

**加权计算：**
```
B1_Score = (84 × 0.40) + (52 × 0.25) + (68.4 × 0.20) + (80 × 0.15)
B1_Score = 33.6 + 13.0 + 13.68 + 12.0
B1_Score = 72.28
```

### 6.4 定性数据编码方案

#### 6.4.1 编码体系

**一级编码（开放式）：**
- 逐行阅读访谈记录
- 提取关键概念，赋予初始代码
- 使用受访者原话或近义表述

**二级编码（轴心式）：**
- 将一级编码归类到主题类别
- 建立类别之间的关系

**三级编码（选择式）：**
- 提炼核心类别
- 构建理论框架

#### 6.4.2 编码框架

| 核心类别 | 子类别 | 代码示例 |
|----------|--------|----------|
| **痛点** | 效率痛点 | #重复劳动 #上下文切换 #搜索困难 |
| | 协作痛点 | #知识孤岛 #标准不一 #沟通成本 |
| | 质量痛点 | #代码质量 #安全顾虑 #维护困难 |
| **需求** | 功能需求 | #模板复用 #多语言 #工具集成 |
| | 体验需求 | #响应速度 #易用性 #稳定性 |
| | 服务需求 | #技术支持 #培训 #定制开发 |
| **行为** | 使用行为 | #高频使用 #特定场景 #多工具并用 |
| | 决策行为 | #个人决策 #团队决策 #采购流程 |
| **态度** | 产品态度 | #满意 #不满意 #期待 #担忧 |
| | 付费态度 | #价格敏感 #价值认可 #预算限制 |

#### 6.4.3 编码示例

| 原话摘录 | 一级编码 | 二级编码 | 三级编码 |
|----------|----------|----------|----------|
| "每次新建项目都要重复配置，很烦" | 重复配置 | 重复劳动 | 效率痛点 |

### 附录B：优先级框架参考

#### B.1 RICE评分法

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

| 维度 | 说明 | 评分标准 |
|------|------|----------|
| Reach | 影响用户数 | 季度预估用户数 |
| Impact | 影响程度 | 3=巨大，2=高，1=中，0.5=低，0.25=极低 |
| Confidence | 信心度 | %表示，需有数据支撑 |
| Effort | 投入成本 | 人月数 |

#### B.2 MoSCoW法则

| 优先级 | 定义 | 决策标准 |
|--------|------|----------|
| Must have | 必须有 | 没有就不发布 |
| Should have | 应该有 | 重要但可妥协 |
| Could have | 可以有 | 有则更好 |
| Won't have | 不会有 | 明确放弃 |

#### B.3 Kano模型

| 属性类型 | 用户态度 | 产品策略 |
|----------|----------|----------|
| 基本型 | 没有会不满，有也不会满意 | 必须满足 |
| 期望型 | 有则满意，没有则不满 | 越多越好 |
| 兴奋型 | 没有无所谓，有则惊喜 | 差异化亮点 |
| 无差异 | 有无都不在意 | 不做 |
| 反向型 | 有反而不满 | 避免 |
